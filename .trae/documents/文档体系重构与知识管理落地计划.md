### 阶段 1：文档体系结构化重构 (Documentation Refactoring)
**目标**：解决文档散乱、层次不清的问题，建立清晰的“文档树”。

1.  **创建标准文档目录结构** (`/docs`)
    *   `/docs/01-architecture`: 架构设计、分层规范、核心流程图。
    *   `/docs/02-guides`: 开发者指南（环境搭建、调试、测试）。
    *   `/docs/03-api`: 核心接口规范、协议说明。
    *   `/docs/04-adr`: 架构决策记录 (ADR) —— 记录**为什么**这么做（如引入 ZString 的决策）。
    *   `/docs/05-business`: 业务规则说明（如消除规则、炸弹逻辑）。

2.  **迁移与拆分现有文档**
    *   将 `ARCHITECTURE.md` (根目录) 迁移至 `/docs/01-architecture/overview.md`。
    *   将 `docs/ARCHITECTURE_GUIDE.md` 拆分为：
        *   `coding-standards.md` (代码规范)
        *   `core-patterns.md` (核心模式：DOD, Pools)
    *   在根目录保留极简的 `README.md`，作为导航入口。

### 阶段 2：自动化文档工具链 (Automation Toolchain)
**目标**：确保文档与代码同步，减少人工维护成本。

1.  **引入 DocFX**
    *   配置 `docfx.json`，用于从 C# XML 注释自动生成 API 文档。
    *   这将解决“API文档维护滞后”的痛点。
2.  **配置 Mermaid 支持**
    *   在 Markdown 中使用 Mermaid 绘制动态架构图和流程图（如消除循环流程）。

### 阶段 3：规范治理与变更追踪 (Governance)
**目标**：让文档成为“活”的资产。

1.  **建立 ADR 机制**
    *   创建 ADR 模板 (`/docs/04-adr/template.md`)。
    *   记录本次“引入 ZString”的决策作为第一个 ADR 实例。
2.  **更新项目规则**
    *   在 `project_rules.md` 中增加文档维护的约束（如“修改核心逻辑必须更新对应业务文档”）。
